# Analysis Strategy Reference

## 表分类：业务表 vs 噪音表

不依赖固定规则，而是通过多维度信号综合判断：

### 噪音信号（倾向跳过）
- **命名特征**：含 log/audit/trace/history/changelog/migration/flyway/liquibase/qrtz/tmp/temp/cache 等
- **结构特征**：列少（<5）+ 行多（百万级）— 典型日志模式
- **功能特征**：系统框架表、权限表、调度表、消息队列表、数据同步表

### 业务信号（倾向分析）
- **实体特征**：列多（>10）、有主键、有索引、命名反映业务概念
- **关联特征**：被多表引用（作为 FK target）、含多个 GUID/ID 关联字段
- **数据特征**：行数适中或增长明显、含日期/状态/金额等业务字段

### 判断原则
- 命名规则因项目而异 — 不要仅靠表名前缀做硬判断
- 结合列名、列数、行数、索引情况综合判断
- 不确定时归入"轻量分析"而非跳过

## 分层分析策略

### Tier 1 — 深度分析（5-10 张核心表）

选择标准：行数 Top + 列数多 + 索引丰富 + 命名指向核心实体

执行：
1. `get_table_detail` — 完整结构：列定义、PK、FK、索引、约束
2. `sample_table_rows`（limit=5）— 真实数据模式
3. `get_table_relationships`（depth=1）— 直接关联

从中提取：
- 主键策略（自增 / UUID / 业务编号）
- 跨表关联模式（FK 字段命名规律）
- 编码/枚举候选列（varchar + 短长度 + 低基数）
- 敏感数据字段（身份证、手机号、银行卡等）
- 时间字段模式（创建/修改/审核/提交等生命周期节点）

### Tier 2 — 轻量分析（10-20 张次要表）

选择标准：中等数据量的业务表、关系表、配置表

执行：
1. `get_table_detail` — 仅看结构

从中提取：
- 表的业务角色（主表 / 从表 / 关联表 / 配置表）
- 与 Tier 1 表的关联关系
- 业务域归属

### Tier 3 — 识别（其余业务表）

不调用工具，根据表名和已知上下文推断用途、归入业务域。

## 业务域发现

不预设域名，从数据中自下而上发现：

1. **前缀聚类** — 按表名前缀（首个 `_` 前的部分）统计表数量
2. **语义归类** — 将相近前缀归为同一业务域（如 `order_`、`order_item`、`order_log` → 订单域）
3. **关联验证** — 通过 FK/GUID 关联确认域间边界
4. **命名生成** — 用中文为每个域命名，反映其业务本质

域名应反映业务语义而非技术实现 — 例如用"交易域"而非"order_前缀表组"。

## 多维观察清单

分析过程中持续关注以下维度，这些发现将写入注解和报告：

| 维度 | 关注点 |
|------|--------|
| 数据规模 | 表/行数量级分布、数据增长热点 |
| 关联模式 | PK/FK 策略、逻辑关联字段命名规律、是否有物理外键 |
| 编码体系 | 疑似枚举/码值字段（短 varchar + 低基数）、分类编码层级 |
| 时间维度 | 创建/修改/审核/归档等时间字段、数据生命周期 |
| 敏感数据 | PII 字段（姓名/证件号/手机/地址）、财务字段 |
| 技术债务 | 冗余表（_tmp/_bak/_old）、历史遗留模式、不一致命名 |
| 数据质量 | 高 NULL 率列、疑似废弃列、异常数据模式 |
