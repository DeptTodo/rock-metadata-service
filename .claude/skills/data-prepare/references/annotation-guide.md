# Annotation Writing Guide

## 为什么注解很重要

注解是将分析过程中获得的**隐性知识**固化为**可检索的结构化元数据**。好的注解能让一个从未接触过这个数据库的人在几分钟内建立全面理解，也能帮助后续的治理工作（数据字典、质量规则、影响分析）做出更好的决策。

因此注解的核心不是"写够字数"，而是"传递最大信息量"。

## 写作原则

### 信息密度

空洞的描述浪费读者注意力。每句话都应该携带具体信息。

对比感受一下：
- 低密度："这是一个重要的业务库，包含很多关键数据"
- 高密度："共 1620 张表、3040 万行，按表名前缀划分为 9 个业务域，核心实体通过 varchar(50) GUID 字段逻辑关联"

两者字数差不多，但第二种让读者立刻建立了具体认知。

### 结构化组织

人在扫描长文本时依赖视觉锚点。用分类标记和层级结构让信息可快速定位。

推荐模式：
```
{一句话定位全局}

【标记A】
- 具体内容

【标记B】
- 具体内容
```

常用标记（按需选用）：【架构概览】【业务域分布】【技术特征】【数据规模】【关联模式】【质量观察】【安全关注】

### 多维覆盖

不同读者关注不同维度。好的注解尽量覆盖多个角度：

| 维度 | 数据源层 | Schema 层 | 表层 |
|------|---------|-----------|------|
| 业务含义 | 系统定位与服务对象 | 模块职责与业务边界 | 实体角色与业务流程位置 |
| 数据规模 | 总量与分布概览 | 域内表/行数分布 | 行数、列数、增长特征 |
| 结构特征 | Schema 划分策略 | 表前缀分组与命名规范 | 列的业务分组与关联字段 |
| 技术细节 | PK/FK 整体策略 | 索引热点、特殊设计 | 具体索引、约束 |
| 关联关系 | 跨 Schema 关联 | 域间依赖 | 与哪些表通过什么字段关联 |
| 质量/风险 | 整体健康度 | 遗留/冗余模式 | 敏感列、编码列、高空值列 |

不需要每个维度都写——没有发现就不写。但在分析中观察到的信息都值得记录。

### 领域自适应

从数据本身推断业务术语。表名叫 `patient` 就用"患者"，叫 `order_item` 就用"订单明细"。如果不确定，保留原始命名并标注推断依据（比如"根据列名 projectname/projectno 推断为项目信息表"）。

这样做的原因是：预设行业模板会导致错误归类，而从数据出发的推断虽然可能不完美，但至少是诚实的。

## 各层级注解要求

### 数据源（update_datasource）

**name**：有意义的中文业务名称。用户会在数据源列表中看到这个名字，需要一眼就能识别。

**description**：这是信息量最大的字段，应当覆盖：
1. **系统定位** — 这个数据库服务于什么业务系统、什么组织
2. **架构概览** — Schema 数量与划分逻辑，每个 Schema 的定位与规模
3. **业务域全景** — 所有发现的业务域，每个域的表数量和核心职责
4. **技术特征** — PK/FK 策略、命名规范、字段编码模式
5. **数据规模** — 总量指标和数据量分布特征
6. **关键发现** — 敏感数据、质量隐患、治理建议

### Schema（update_schema_attrs）

**displayName**：清晰的中文业务名称。

**businessDescription**：覆盖：
1. **定位** — 在整个系统中的角色
2. **域明细** — 每个业务域的核心表及职责（表名 + 列数 + 行数 + 一句话说明），这是最有价值的部分，帮助读者快速定位感兴趣的表
3. **技术亮点** — 索引热点、PK 模式、数据量热点表

### 核心表（update_table_attrs）

只对 Phase 3 中 Tier 1 分析过的表做注解（通常 5-10 张），因为只有深度分析过的表才有足够信息写出高质量注解。

**displayName**：中文业务名称。

**businessDescription**：覆盖：
1. **业务角色** — 在业务流程中的位置
2. **列群分析** — 按业务含义对列分组（如"项目基本信息组、资金来源组、审核流程组"），这比逐列描述更有价值，因为核心表通常有上百列
3. **关联关系** — 通过什么字段与哪些表关联
4. **数据观察** — 数据量级、编码列、敏感列

**businessDomain**：Phase 3 中发现的业务域名称。

**importanceLevel**：
- `CORE` — 核心业务实体的主表（每个域通常只有 1-3 张）
- `IMPORTANT` — 重要业务表、高频引用的关系表
