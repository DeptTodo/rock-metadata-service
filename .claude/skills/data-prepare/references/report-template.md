# Report Template

Phase 5 结束时输出的结构化报告模板。根据实际数据填充，每个区块的具体呈现方式应适配发现的内容。

```markdown
## 数据治理准备报告

### 数据源概况
- 名称: {业务化名称}
- 类型: {dbType} {version}
- 连接: {host}:{port}/{databaseName}
- Schema: {N} 个（应用 {N} / 系统 {N}）
- 表: {N} 张 | 列: {N} 个 | 总数据量: 约 {N} 万行

### Schema 架构
| Schema | 类型 | 表数量 | 数据量 | 定位 |
|--------|------|--------|--------|------|
| ... | 应用/系统 | ... | ... | 一句话说明 |

### 业务域全景
| 业务域 | 表前缀 | 表数量 | 核心表 | 说明 |
|--------|--------|--------|--------|------|
| {域名} | {prefix_} | {N} | {核心表1}, {核心表2} | 业务职责 |

### 核心表概览（按数据量 Top N）
| 表名 | Schema | 行数 | 列数 | 业务说明 |
|------|--------|------|------|----------|
| ... | ... | ... | ... | ... |

### 技术特征
- **主键策略**: {描述}
- **关联模式**: {描述}
- **命名规范**: {描述}
- **索引热点**: {描述}

### 已完成注解
| 层级 | 数量 | 说明 |
|------|------|------|
| 数据源 | 1 | name + description |
| Schema | {N} | displayName + businessDescription |
| 核心表 | {N} | displayName + businessDescription + domain + importance |

### 关键发现
- {发现1：数据特征/质量观察/安全关注/架构特点}
- {发现2}
- ...

### 跳过的表（{N} 张）
按类别汇总：{日志类 N 张}, {系统框架类 N 张}, {临时表 N 张}, ...

### 后续建议
1. {优先级最高的治理动作}
2. {次优先级}
3. ...

常见后续方向：
- 对核心表的列级注解（displayName + businessDescription + sensitivityLevel）
- 发现的编码/枚举列 → 数据字典建设（data-dict skill）
- 敏感字段标记与分级（sensitivityLevel + maskingStrategy）
- 数据质量规则定义（quality rules）
```

## 报告写作要点

- **数字具体** — 用实际数据而非模糊描述（"1620张表" 而非 "大量表"）
- **层次清晰** — 从全景到局部逐步展开
- **可执行** — 后续建议要具体，指向明确的下一步动作
- **透明** — 跳过了什么、为什么跳过，让用户了解分析的覆盖范围
